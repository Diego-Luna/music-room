# Phase 8 — PLAYLIST rooms (fractional indexing)

## Scope

Cover **V.2.3 Music Playlist Editor** end-to-end:
- Visibility (PUBLIC / PRIVATE)
- License: everyone-edits vs invited-only-edits
- **Concurrent edit race conditions** (subject explicitly flags this)

## The concurrency problem

Subject:
> "You should especially care about the management of competition
> problems: for instance, if several people move different tracks
> or the same one in a playlist."

Naive solutions all break under concurrent edits:

| Strategy                         | Failure mode                            |
|----------------------------------|------------------------------------------|
| `position: Int` reindexed on move | Concurrent moves overwrite each other's renumber; the playlist scrambles |
| `position: Float` (midpoint between neighbors) | Floats lose precision after ~50 moves between the same pair |
| Linked list (`previousId` per item) | Single-row contention on every move; deadlocks under load |
| Distributed CRDT (RGA, Yjs…)     | Way too heavy for a 42 project          |

## What we picked: fractional indexing

`position: String` storing a fractional key generated by the
`fractional-indexing` library (`generateKeyBetween(prev, next)`).

```
position "a0"          ← first item
position "a1"          ← item moved between "a0" and end
position "a0V"         ← item moved between "a0" and "a1"
position "a0VV"        ← item moved between "a0" and "a0V"
position "a0VVV"       ← still works after thousands of inserts at the same pair
```

The key is a base-62 string. Comparing two keys lexicographically
gives the correct order. **Between any two distinct keys, an
infinite number of keys can be inserted** — there's never a
"renumber the playlist" pass.

`generateKeyBetween(a, b)` returns a key strictly between `a` and `b`,
shorter than the predecessor when possible.

## Why concurrent moves can't conflict

Two users move items A and B to the same position simultaneously:
- User 1: `move A between X and Y` → A.position = `generateKeyBetween(X.pos, Y.pos)`
- User 2: `move B between X and Y` → B.position = `generateKeyBetween(X.pos, Y.pos)`

Both inserts succeed independently because:
1. They write **different rows** (A and B), so no row-lock contention.
2. Both new positions sit between X and Y in lex order, just at
   different "slot widths". The resulting playlist:
   `... X, A, B, Y, ...` or `... X, B, A, Y, ...` (whichever wrote
   first), but **never** `... X, Y, A, B, ...` (impossible since
   A.pos < Y.pos by construction).

No transaction needed for move operations. No deadlock. No "playlist
got scrambled" bug.

The `loadtest/03_playlist_reorder.js` script runs 20 concurrent
editors doing random moves for 45 s and verifies the final state is
a valid lex-sorted permutation.

## License: everyone-edits vs invited-only-edits

Subject:
> "By default, everyone can edit the playlist.
> With the right license, the invited users are the only ones who
> can edit the playlist."

Schema: `Room.allowMembersEdit: Boolean @default(true)`.

`PlaylistService.requireEditor(room, userId)`:
- Owner always allowed
- Else find the `RoomMember` row for `(roomId, userId)`
- If `allowMembersEdit === true` → any member (including MEMBER role)
- If `allowMembersEdit === false` → only OWNER/ADMIN

The `allowMembersEdit` flag is updatable via `PATCH /rooms/:id` by
the owner — the "right license" in subject vocabulary.

## Operations and endpoints

```
GET    /rooms/:id/playlist                  list ordered items (lex sort on position)
POST   /rooms/:id/playlist                  add a track; can specify afterTrackId / beforeTrackId
PATCH  /rooms/:id/playlist/:trackId/move    move via afterTrackId / beforeTrackId references
DELETE /rooms/:id/playlist/:trackId         remove (author / owner / admin)
```

`POST` and `PATCH` accept either `afterTrackId` or `beforeTrackId`
(not both, validated at the controller). The service computes the
new position as `generateKeyBetween(after.pos, before.pos)`.

## Soutenance defense points

- *Why fractional indexing instead of integer reindexing?* —
  Reindexing forces a transaction that touches every row on every
  move, which scales O(N) per write. Under 20 concurrent editors
  the contention is unworkable. Fractional indexing makes every
  move a single-row UPDATE.
- *Why a string-based fractional key and not a float?* — Floats
  lose precision after ~50 moves into the same gap. Strings can
  grow indefinitely without precision loss.
- *Does this scale to 100 editors?* — Yes, because each move
  is a single-row write with no read-modify-write cycle on
  neighbors. The DB row lock is held only for the duration of one
  UPDATE.
- *What if two clients try to insert at the exact same position?* —
  They get different fractional keys (the function takes the
  predecessor's value, not a global lock). Both inserts succeed.
  The order in the resulting playlist is determined by which write
  reached Postgres first.
- *How is "invited only edit" different from "private room"?* —
  Private room gates **who can see the playlist** (visibility);
  invited-only-edit gates **who can mutate** it. A public playlist
  with `allowMembersEdit=false` is a "broadcast" mode where anyone
  can listen but only the owner/admins curate.
- *Who can delete an item?* — The author of the item, the room
  owner, or any room ADMIN. Encoded in
  `PlaylistService.removeItem()`.
